The Pluton Framework: Service Keys

Service Keys are the mechanism by which clients address services, configuration files are identified and various temporary file system objects are created (eg, Rendezvous Sockets). Service Keys consists of dot separated set of components, i.e.: 
	Application.Function.Version.Serialization
Historically the naming is RPC-oriented, but really, Application and Function can just as easily be Action and Resource - especially if you take advantage of the lookup wildcard feature. In that case, a Service Key could be viewed as: 
        Action.Resource.Version.Serialization

The rules which define a valid Service Key are:

  • The Service Key must have exactly four components separated by a '.'

  • Each component must be less than or equal to 32 characters in length - mostly to ensure the complete name plus extensions can be created in a Unix file system.

  • The first component, called Application must have a length which is greater than zero and less than or equal to 32 characters.

  • The second component, called Function must have a length which is less than or equal to 32 characters. Function can be zero length for a configuration file to indicate a wild-card service, but must be greater then zero when used by a client to identify a service.

  • The third component, called Version must be a positive decimal integer with a length in the range of 1 to 6 characters. To avoid ambiguity, leading zeroes are not allow for versions greater than zero as this could be mis-interpreted by some to mean octal.

  • The forth and final component, called Serialization, must exactly match one of the pre-defined serialization types.
